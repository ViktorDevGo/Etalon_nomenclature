# Отчет: Почему отсутствуют диски за 06-08.03.2026

## 🔍 Проблема

В БД обнаружено:
- **Шины**: 94,561 записей (06.03 - 09.03)
- **Диски**: 12,835 записей (только 09.03) ❌
- **Ожидалось**: ~22,700 дисков (06.03 - 09.03)

Отсутствуют диски за 06-08.03.2026 (~10,000 записей)

## 🕵️ Причина (Root Cause Analysis)

### Хронология событий:

**06.03.2026 - 07.03.2026**
- Письма обработаны старым кодом
- ❌ Disk parser еще НЕ существует
- ✅ Шины записаны в price_tires
- ✅ Email помечен как processed
- ❌ Диски НЕ записаны (парсера нет)

**08.03.2026 16:14**
- 🎯 Commit 6c435c9: "Add disk price handling and parsing capabilities"
- ✅ Disk parser добавлен в код

**09.03.2026 ~14:24**
- Код задеплоен на production
- ✅ Новые письма (09.03) обработаны с дисками
- ❌ Старые письма (06-08.03) УЖЕ в processed_emails → пропущены!

### Почему атомарность не помогла?

Атомарность работает правильно! Но:
- Письма за 06-08.03 были обработаны **ДО** добавления disk parser
- На тот момент парсился только один тип данных (шины)
- Транзакция прошла успешно (шины + email mark), диски просто не парсились

## ✅ Решение

### 1. Унификация названия поставщика

Везде в коде используется "ГРУППА БРИНЕКС", но в БД есть старые записи с "БРИНЕКС".

**Файл:** `unify_provider_names.sql`
```sql
UPDATE price_disks SET provider = 'ГРУППА БРИНЕКС' WHERE provider = 'БРИНЕКС';
UPDATE price_tires SET provider = 'ГРУППА БРИНЕКС' WHERE provider = 'БРИНЕКС';
```

### 2. Повторная обработка писем за 06-08.03

**Файл:** `reprocess_06_08_emails.sql`
```sql
-- Удалить записи из processed_emails за 06-08.03
DELETE FROM processed_emails
WHERE DATE(email_date) BETWEEN '2026-03-06' AND '2026-03-08';
```

После выполнения:
- Письма перейдут в статус "необработанные"
- Следующий цикл (макс 1 минута) обработает их заново
- ✅ Шины обновятся (дедупликация по article)
- ✅ Диски запишутся ВПЕРВЫЕ
- ✅ Email снова пометится как processed
